## 摘要
近似最近邻搜索（ANNS）是现代人工智能系统中的核心原语，其中基于图的方法目前在大规模场景下提供了最佳的精度-效率权衡。该工作负载本质上是内存受限的：图遍历会产生频繁、不规则的内存访问，使得CPU吞吐量受限于主内存带宽，而GPU缺乏支持十亿级索引的高带宽内存容量。处理进内存（PIM）是一个自然的选择，因为将计算放置在数据旁边可以释放此类带宽匮乏工作负载所需的大量内部带宽。然而，将基于图的ANNS移植到PIM会暴露出几个架构不匹配问题：每个处理单元只有很小的本地内存，单元间通信成本高，主机协调增加开销，且内存内计算单元相对较弱——这些限制迫使先前的基于PIM的ANNS设计退回到基于聚类的索引，其召回率上限远低于图方法。本文提出了一种算法-架构协同设计，通过三个组件克服了这些障碍：紧凑的索引布局，将PIM驻留内存占用缩小14.5倍；异步流水线调度器，保持主机到PIM互连饱和；以及无乘法距离内核，召回率损失低于0.08%。在三个十亿级基准测试中，所提出的设计相比CPU和GPU基线分别实现了高达20倍和17.1倍的吞吐量提升，在高召回率区域相比先前的PIM加速器提升了129倍，并且在多节点部署和新兴PIM架构中优雅地扩展。

## Abstract
Approximate Nearest Neighbor Search (ANNS) is a core primitive in modern AI systems, and graph-based methods currently offer the best accuracy-efficiency trade-off at scale. The workload is fundamentally memory-bound: graph traversal produces frequent, irregular memory accesses that cap CPU throughput at main-memory bandwidth, while GPUs lack the high-bandwidth memory capacity to host billion-scale indexes. Processing-in-Memory (PIM) is a natural candidate, as placing computation next to data unlocks the abundant internal bandwidth that such bandwidth-starved workloads demand. Porting graph-based ANNS to PIM, however, exposes several architectural mismatches: each processing unit has only a small local memory, inter-unit communication is costly, host coordination adds overhead, and in-memory compute units are relatively weak -- limitations that have forced prior PIM-based ANNS designs to fall back on cluster-based indexing, whose recall ceiling is far below that of graph methods. This paper presents an algorithm-architecture co-design that overcomes these obstacles through three components: a compacted index layout that shrinks the PIM-resident memory footprint by 14.5x; an asynchronous pipelined scheduler that keeps the host-to-PIM interconnect saturated; and a multiplication-free distance kernel that loses under 0.08% recall. Across three billion-scale benchmarks, the proposed design achieves up to 20x and 17.1x higher throughput than CPU and GPU baselines, respectively, outperforms prior PIM accelerators by 129x in the high-recall regime, and scales gracefully across multi-node deployments and emerging PIM architecture.

### 1. 论文的核心问题与整体含义（研究动机和背景）

- **核心问题**：近似最近邻搜索（ANNS）是现代AI系统（如检索增强生成RAG、推荐系统）的基础原语。基于图的方法（如SymphonyQG）在十亿规模下提供了最佳精度-效率权衡，但该工作负载本质上是**内存受限**的：CPU受限于主存带宽（如375 GB/s），GPU虽带宽更高（A100：2.0 TB/s）但HBM容量不足以容纳十亿级图索引（如SymphonyQG索引超过1.25TB）。处理进内存（PIM）架构将计算嵌入内存附近，提供巨大内部带宽（如UPMEM合计2.8 TB/s），似乎是理想方案。
- **挑战**：直接移植图ANNS到PIM面临四个紧密耦合的架构不匹配：
  1. **极端本地内存容量**：每个PU仅有64 MB（UPMEM），而十亿级图索引超1.25 TB，需激进分区。
  2. **跨PU通信昂贵**：细粒度分区导致频繁跨PU遍历，外部带宽比内部慢10倍以上。
  3. **协调开销与负载不均**：批同步调度导致空闲等待，逐查询调度碎片化通信带宽。
  4. **PU计算能力受限**：PIM核心算术能力弱，甚至缺乏乘法器（如UPMEM），量化距离核中的浮点乘除法开销巨大。
- **整体意义**：现有PIM ANNS方案（如UpANNS、PIMANN）仅能实现基于聚类的低质量算法（召回率上限约61%），无法达到图方法的召回率。本文通过算法-硬件协同设计，首次在商用PIM上高效实现高召回率的十亿级图ANNS，填补了架构鸿沟。

### 2. 论文提出的方法论：核心思想、关键技术细节

论文提出**PIMCQG**框架，包含三项协同创新：

#### (1) PIM友好的紧凑索引（解决C1、C2）
- **消除冗余**：原始SymphonyQG按边存储量化码和因子，同一邻居节点被重复编码。采用IVF风格聚类，以聚类中心为参考量化每个节点一次（单套RabitQ码/因子），图邻接表仅存邻居ID，PU遍历时通过ID索引规范码。此举消除了边冗余。
- **移除原始向量**：将全精度原始向量从PIM中移除，仅保留在主机端。PIM内仅做近似距离遍历，主机端对PIM返回的候选集进行精确重排序。
- **效果**：索引大小降低14.5×（如SSN1B从2385 GB降至164 GB），使十亿级图可放入PIM分布式内存，并缓解跨PU通信压力。

#### (2) 异步流水线调度（解决C3）
- **离线聚类放置**：基于访问频率的贪心负载均衡将聚类分配到PU，结合紧凑索引的内存节省获得调度灵活性。
- **动态小批量**：主机维护每PU输入缓冲，当待发查询达到阈值或等待超时后触发小批量发送，平衡通信效率与响应性。
- **五阶段异步流水线**：主机准备→主机→PIM传输→PIM内搜索→PIM→主机返回→主机重排序。这些阶段通过FIFO解耦并重叠执行，避免全局屏障，使通信、搜索、后处理并行。
- **小批量大小优化**：根据T(N_B)=max(T_pre,T_proc,T_post)/N_B 选择最优N_B，使各阶段耗时均衡，数据大小保持在高效通信区间。

#### (3) 无乘法距离计算（解决C4）
- **简化公式**：原始RabitQ距离核中，标量缩放因子1/α = 1/cos(θ) 需要浮点乘法。观察发现同一IVF聚类内cos(θ)经验稳定（约为0.8），因此用聚类级常数α代替节点级因子。
- **位运算实现**：设置α=0.8，得1/α=1.25=1.01₂，通过移位和加法实现：x·1.25 ≈ x + (x>>2)。对于偏离默认值的聚类，在索引构建时校准为最近的硬件友好二进制移位值。
- **精度**：固定α导致召回率下降<0.08%（图9）。
