## 摘要
移动设备上运行大型语言模型（LLM）的发展受隐私和快速响应时间需求的驱动。移动设备上高能耗的数据传输使得存内计算（PIM）成为一种有效的解决方案。由于严格的DRAM成本约束、电路板上有限的物理面积以及应用与LLM之间的交互，CPU和PIM必须在共享内存空间内并发运行。然而，诸如存储体冲突和总线拥塞等挑战可能会出现，可能削弱PIM的性能和能效优势。为了应对这一挑战，我们提出了COSM，这是一种协同调度框架，旨在促进移动平台上PIM和CPU任务的并发执行。我们的关键创新包括：1）一种低干扰的PIM控制接口，在不中断CPU内存访问的前提下生成最大数量的PIM命令；2）一种空闲感知调度方法，将PIM命令整合到CPU访问序列中的可用空闲时间窗口内。COSM不仅对CPU隐藏了PIM执行延迟，还将PIM执行与数据传输重叠。在并发执行LLM和移动工作负载（包括移动应用和计算密集型内核）的实验表明，与基线调度方法相比，COSM将PIM吞吐量提升高达2.8倍，而CPU性能损失小于2.0%。

## Abstract
The development of on-device large language models (LLMs) is driven by the need for privacy and fast response times. Energy-intensive data transfer on mobile devices makes Processing-in-Memory (PIM) an effective solution. Due to stringent DRAM cost constraints, limited physical footprint on circuit boards, and the interaction between applications and LLMs, it is imperative for the CPU and PIM to operate concurrently within a shared memory space. However, challenges such as bank conflicts and bus congestion can arise, potentially diminishing the performance and energy benefits of PIM. To address this challenge, we introduce COSM, a cooperative scheduling framework designed to facilitate the concurrent operation of PIM and CPU tasks on mobile platforms. Our key innovations include: 1) a low-interference PIM control interface that generates the maximum number of PIM commands without disrupting CPU memory accesses; 2) an idleness-aware scheduling method that integrates PIM commands into available idle time windows within the CPU's access sequence. COSM not only hides PIM execution latency from the CPU, but also overlaps PIM execution with data transfer. Experiments on concurrent execution of LLMs and mobile workloads, including mobile applications and compute-intensive kernels, demonstrate that COSM improves PIM throughput by up to 2.8x compared to the baseline scheduling method with less than 2.0% CPU performance loss.

## 1. 核心问题与整体含义（研究动机和背景）
- **动机**：移动设备部署大语言模型（LLM）对实时性和隐私有强烈需求，但传统CPU/NPU架构面临数据移动能耗高（超过60%总能耗）和外部带宽瓶颈（LPDDR5X通常<80 GB/s），导致热问题和响应延迟。
- **背景**：存内计算（PIM）通过将计算单元集成在DRAM内部，利用内部高带宽（102.4 GB/s）绕过外部带宽瓶颈，能显著降低功耗（Samsung LPDDR5-PIM实测70%能耗降低）。然而，由于移动设备DRAM成本约束和物理面积限制，CPU与PIM必须共享同一物理内存空间，并发执行时会产生bank冲突、命令总线拥塞和CPU-mediate传输干扰，严重削弱PIM的能效优势。现有调度方法（如CPU优先、行命中感知）要么牺牲PIM性能，要么严重损害CPU延迟，缺乏平衡方案。

## 2. 方法论：核心思想、关键技术细节
- **核心思想**：利用CPU工作负载对内存访问的“短暂空闲窗口”——包括bank级的空闲（请求间间隔）和总线级的空闲（激活-数据访问之间的间隙），将PIM命令插入这些空闲时段，实现低干扰并发执行。
- **关键创新一：低干扰PIM控制接口**
  - **可抢占PIM执行命令**：引入 `PIM_Exec`（执行指令）和 `PIM_Pause`（暂停指令）。PIM单元可连续执行最多`nPTL`个列的运算，无需主机频繁触发；当CPU请求到达时，`PIM_Pause`迫使PIM在列边界中断，保存状态（列状态计数器、命令寄存器），CPU访问后通过“恢复”命令从断点继续。这消除了固定长度命令中长命令阻塞CPU、短命令饱和命令总线的矛盾。
  - **带宽解耦数据传输命令**：将传统的CPU-mediate传输拆分为两阶段：`PIM_WrBuf/PIM_RdBuf`（外部总线传输，占用外部带宽）和`PIM_LdBuf/PIM_StBuf`（内部bank与buffer间传输，占用内部带宽）。两者可独立调度，且内部传输同样可被抢占（与`PIM_Exec`共享`nPTL`长度）。这打破了传统传输必须连续锁定总线的限制，允许利用零散的空闲窗口。
- **关键创新二：空闲感知内存调度（Idleness-aware Scheduling）**
  - **空闲窗口估计器（IWE）**：通过分析CPU请求队列，采用改进的FR-FCFS模拟，预测每个bank的最早服务时间，从而得到bank级和总线级空闲窗口。具体算法：根据bank状态（行关闭、已打开目标行、已打开非目标行）计算请求就绪周期，按rank优先和行命中优先规则模拟，输出每个bank的`service_time`，进而得到`window_bank`和`window_bus`。
  - **PIM调度器**：基于IWE信息，对`PIM_Exec`、`PIM_LdBuf/StBuf`、`PIM_WrBuf/RdBuf`三级命令执行优先级：优先使用总线空闲窗口发送外部传输命令（`PIM_WrBuf/RdBuf`），其次确保buffer不阻塞（发送`PIM_LdBuf/StBuf`清空缓冲区），最后将`PIM_Exec`插入bank空闲窗口。通过将PIM工作负载划分为独立tile（如子矩阵），实现“一个tile的数据传输+另一个tile的执行+第三个tile的结果收集”三级重叠流水。
  - **命令仲裁器**：严格遵循CPU优先原则：若IWE预测PIM操作将延迟CPU请求，则立即发送`PIM_Pause`；否则，优先服务CPU请求，仅在CPU队列空或总线/银行空闲时插入PIM命令。

## 6. 主要结论与发现
- COSM在并发执行LLM与CPU负载时，PIM吞吐量相比Chopim提升2.8倍，相比All-Bank提升6.0倍；CPU性能损失仅2.0%（平均），远低于AsyncDIMM-Bank的89.9%损失。
- 可抢占PIM执行命令消除了固定长度命令中长命令阻塞CPU、短命令饱和总线的折中，在CPU性能损失<3.2%的条件下达到最大PIM性能，相比32周期固定命令提升2.02倍。
- 带宽解耦+空闲感知调度将CPU性能从Base配置的平均下降约40%降低到<5%，主要通过将传输操作限制在总线空闲窗口内。
- 空闲感知调度相比CPU优先调度，额外利用37%的可用内部带宽，PIM性能提升21%。
- 能耗方面，COSM比AsyncDIMM-Bank低1.34倍，比Chopim低1.61倍（每token能耗）。
- 面积开销仅0.069 mm²（7.4%），主要来自PIM调度器、IWE和命令仲裁器。

## 7. 优点
- **接口与调度协同设计**：可抢占命令和带宽解耦传输从根本上解决了PIM命令的细粒度与效率矛盾，为调度提供了灵活控制基础。
- **全局视角的空闲利用**：不仅利用bank间请求间隔，还利用行激活-数据访问之间的“隐藏”间隙（通过延迟ACT实现），提升了内部带宽利用率。

- **低面积开销**：硬件模块在5nm工艺下仅0.069 mm²，具备实际部署可行性。
- **可扩展性**：作者指出COSM的原则可推广至服务器、数据中心等共享内存CPU-PIM系统，以及异构加速器（GPU/NPU）场景。
