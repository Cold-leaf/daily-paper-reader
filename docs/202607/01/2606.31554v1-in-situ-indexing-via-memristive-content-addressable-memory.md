## 摘要
处理中内存（PIM）是一种已被证实的克服“内存墙”的范式。然而，尽管数据索引受到同一内存墙的严重瓶颈，但索引如何有效利用PIM的独特能力仍不清楚。我们提出了PATH，一种原位索引架构，通过利用PIM的大规模并行性和固有的数据移动来弥合这一差距。具体来说，我们首先将基本的索引操作，即插入、搜索、更新和删除，重新表述为直接在内存阵列内执行的高度并行的原位内容可寻址存储器操作。以哈希索引为典型案例，我们详细阐述了PATH如何打破传统哈希方案中内存访问、负载因子和处理延迟之间的固有权衡。通过采用超大逻辑桶和内存内移动，PATH几乎消除了哈希冲突解决的成本，并显著降低了调整大小的开销。与最先进的方案相比，PATH在插入操作下实现了4.7-7.8倍的吞吐量提升、>14.5倍的尾部延迟降低和>61.4%的内存访问减少，为下一代数据中心计算奠定了可扩展的基础。

## Abstract
Processing-in-Memory (PIM) is a proven paradigm for overcoming the ``memory wall". However, while data indexing is severely bottlenecked by this same wall, it remains unclear how indexing can effectively benefit from PIM's unique capabilities. We present PATH, an in-situ indexing architecture that bridges this gap by leveraging the massive parallelism and inherent data-movement of PIMs. Specifically, we first reformulate the fundamental indexing operations, namely Insert, Search, Update, and Delete, into highly parallel in-situ content-addressable memory operations executed directly within memory arrays. Taking hash indexes as a typical case, we elaborate how PATH breaks the inherent trade-off among memory accesses, load factor, and process latency in conventional hashing schemes. By adopting ultra-large logical buckets and in-memory moving, PATH virtually eliminates the cost of hash collision resolution and significantly reduces resizing overhead. Compared with state-of-the-art schemes, PATH achieves $4.7-7.8\times$ higher throughput, $>14.5\times$ lower tail latency, and $>61.4\%$ fewer memory accesses under insertions, laying a scalable foundation for next-generation data-centric computing.

# 基于忆阻内容可寻址存储器的原位索引（PATH）——详细总结

## 1. 论文的核心问题与整体含义（研究动机和背景）

- **背景**：处理中内存（PIM）已被证明是克服“内存墙”（计算速度与内存访问延迟之间的差距）的有效范式。然而，数据索引（如哈希索引、树索引）严重受限于同一内存墙：传统哈希索引需要多次内存访问和比较来处理哈希冲突（碰撞），树索引需要逐节点遍历，都导致大量数据移动和延迟。
- **核心问题**：现有PIM架构（如用于神经网络的PRIME、用于图计算的GraphR）虽具备原位计算能力，但如何让索引操作（插入、搜索、更新、删除，即ISUD）有效利用PIM的并行性和数据移动能力尚不清楚。关键挑战在于索引的结构/操作与PIM硬件特征之间的不匹配。
- **整体目标**：提出PATH架构，首次构建原位（in-situ）索引，通过硬件/软件协同设计（HW/SW co-design）将索引操作重构为内存阵列内的高度并行内容可寻址存储器（CAM）操作，突破传统哈希方案在内存访问、负载因子和延迟之间的固有权衡，为数据中心计算奠定可扩展基础。

## 2. 论文提出的方法论：核心思想、关键技术细节

### 核心思想
- 将索引的ISUD操作重新表述为直接在记忆体阵列内执行的原位CAM操作。利用忆阻ReCAM（忆阻内容可寻址存储器）阵列的大规模并行性：在一个ReCAM阵列中，可同时对整行（如512行）进行并行比较，从而大幅减少内存访问次数。
- 通过硬件/软件协同设计，针对哈希索引（作为代表性案例）提出新技术来消除碰撞解决和重排开销。

### 关键技术细节
1. **原位ISUD抽象**：
   - 每个ReCAM阵列按行存储键和有效标志位（valid flag）。值用普通单元存储（不参与比较）。
   - **插入**：先用并行比较找到有效标志位为0的空行，然后写入键值对和有效标志。
   - **搜索/更新/删除**：用输入键和有效标志1进行并行比较，找到匹配行后，分别进行读取值、写入新值、清除有效标志。
   - 并行性：一个512行的阵列可同时在一次CAM操作中处理512个键的比较。

2. **硬件/软件协同设计**（以哈希索引为例）：
   - **超大型逻辑桶（ultra-large logical buckets）**：每个哈希桶只存储少量指针（地址），指向多个ReCAM阵列（每个阵列可存储512个KV项），从而逻辑上覆盖大量数据。桶本身仅64B对齐，一次内存访问可加载整个桶到CPU缓存。
   - **等待自由插入（Wait-free Insertion）**：在桶内维护元数据（每个关联ReCAM阵列的有效KV计数），若计数小于阵列行数则无需等待插入成功信号，直接通过无等待写入完成，显著降低插入延迟。
   - **被动碰撞解决（Passive Collision Resolution）**：由于每个桶可容纳大量KV项（5×512=2560项），负载因子可达91.8%，因此无需主动碰撞解决策略（如链式探测、开放地址等）。当桶满时直接触发重排（resizing）。
   - **交错数据映射（Interleaved Data Mapping）**：将哈希桶按银行（bank）级别交错映射，使不同银行可同时执行ISUD操作，提升并行性。
   - **规则引导的内存内移动（Rule-guided In-memory Moving）**：针对重排或分裂，利用简单规则（如哈希值的一位指示、与枢轴键的比较）在内存阵列内直接移动数据，避免昂贵的片外内存访问。例如重排时根据hash(key)的某一位决定KV项留在原位置还是移动到新位置。

3. **成本模型对比**：
   - 传统哈希方案：延迟 $T_{conv} \approx T_{hash} + (N_{mem}-1) \times (T_{mem}+T_{cmp}) + (T_{mem}+T_{op})$，其中 $N_{mem} \approx \lceil N_{probe}/n_{conv}\rceil$。
   - PATH方案：延迟 $T_{path} \approx T_{hash} + (N'_{mem}-1) \times (T_{mem}+T_{miss}^{isud}) + (T_{mem}+T_{match}^{isud})$，其中 $N'_{mem} \approx \lceil N_{probe}/n_{path}\rceil$。
   - 由于 $n_{path} \gg n_{conv}$（如512 vs 4-8），因此 $N'_{mem} \ll N_{mem}$，延迟更优。

### 算法流程（文字说明）
- **插入操作**：计算hash(key) → 获取目标桶地址 → 检查桶内元数据找到有空闲slot的ReCAM地址 → 若计数小于阵列行数，执行无等待原位插入（直接写入键值和标志）；若满则触发重排。
- **搜索/更新/删除**：定位桶后，依次读取每个slot对应的ReCAM地址，发送原位操作命令，根据匹配结果进行相应操作。
- **重排（resizing）**：使用规则引导的内存内移动：根据hash(key)的下一有效位（指示位）决定每个KV项新位置（原位置或原位置+N）。硬件控制器支持列读取指示位后逐行移动数据。

- PATH架构通过原位ISUD和HW/SW协同设计，大幅提升索引性能。

## 8. 不足与局限

- **模拟验证**：所有结果基于Gem5模拟，缺乏真实芯片或FPGA原型验证，性能和能耗数据可能与实际有偏差。
- **写入延迟敏感度**：虽然做了一定敏感性分析（40-400ns），但PATH方案的相对优势在写延迟变化时波动不大，仍需更极端场景测试。
- **扩展性限制**：多线程场景下，PATH-EH相对CCEH的加速比不如单线程显著（3.1× vs 4.4×），说明软件并发控制成为新瓶颈。
- **写密集型优势更大，读密集型优势减弱**：在S100%或I5%工作负载下，PATH的吞吐量优势缩小（仅1.3×），且能量消耗略有上升（由于每CAM比较能耗更高）。
- **特定假设**：键值存储假设键为固定长度（如64位），且单值不参与比较；对于变长键或复杂值需要额外设计。
- **树索引未深度优化**：PATH-Tree仅简单替换叶子节点为ReCAM，未像哈希索引那样做全流程HW/SW协同，可能未充分发挥潜力。
- **跨银行移动限制**：内存内移动只限于同一银行内，可能影响数据局部性较差的场景。
- **缺少与真实PIM硬件（如UPMEM）的深度对比**：仅比较了延迟（PIMLex），未比较吞吐量或适用场景。
