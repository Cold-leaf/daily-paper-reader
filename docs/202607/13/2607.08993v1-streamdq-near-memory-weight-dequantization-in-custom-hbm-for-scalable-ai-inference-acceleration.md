## 摘要
随着大语言模型（LLM）规模的扩大，其内存和计算需求大幅增长，权重量化成为降低模型体积且精度损失最小的广泛采用技术。然而，在当前的GPU上，基于CUDA核心的反量化会引入大量指令开销、片上流量和流水线停顿，使其成为高吞吐量、云规模LLM服务的主要瓶颈。为解决这些限制，我们提出StreamDQ，一种轻量级架构增强，可在内存子系统中实现即时反量化，用于高吞吐量、大批量LLM推理。StreamDQ将紧凑的反量化块（DQB）集成到高带宽内存（HBM）的基底芯片中，并在标准内存加载上执行内联反量化。每个内存读取请求上的轻量级边带标签选择反量化模式，同时保留常规加载语义。通过将反量化迁移到内存侧，StreamDQ消除了GPU侧基于CUDA核心的反量化，从而减少GPU上的片上流量，并避免大批量时反量化权重的额外HBM写回和重新加载。我们的评估表明，StreamDQ在混合精度GEMM中实现了高达7.08倍的加速和90.23%的能量降低，每个DQB在12纳米CMOS工艺中仅占用0.127平方毫米面积和0.355瓦功耗开销。对于端到端LLM推理，StreamDQ将延迟降低高达54.68%，并将解码吞吐量提高高达2.20倍。

## Abstract
As large language models (LLMs) scale, their memory and computation demands have grown substantially, making weight-only quantization a widely adopted technique for reducing model size with minimal accuracy loss. However, on current GPUs, CUDA-core-based dequantization introduces substantial instruction overhead, on-chip traffic, and pipeline stalls, making it a major bottleneck for high-throughput, cloud-scale LLM serving. To address these limitations, we propose StreamDQ, a lightweight architectural enhancement that enables on-the-fly dequantization in the memory subsystem for high-throughput, large-batch LLM inference. StreamDQ integrates compact DeQuantization Blocks (DQBs) into the base die of high-bandwidth memory (HBM) and performs inline dequantization on standard memory loads. A lightweight sideband tag on each memory read request selects the dequantization mode while preserving conventional load semantics. By relocating dequantization to the memory side, StreamDQ eliminates GPU-side CUDA-core-based dequantization, thereby reducing on-chip traffic on the GPU and avoiding extra HBM write-back and reload of dequantized weights at large batch sizes. Our evaluation shows that StreamDQ achieves up to 7.08$\times$ speedup and 90.23\% lower energy for mixed-precision GEMM, with only 0.127\,mm$^2$ area and 0.355\,W power overhead per DQB in a 12\,nm CMOS process. For end-to-end LLM inference, StreamDQ reduces latency by up to 54.68\% and improves decode throughput by up to 2.20$\times$.

## 2. 方法论：核心思想、关键技术细节
- **核心思想**：将紧凑的反量化块（DeQuantization Block, DQB）集成到定制 HBM（Custom HBM）的基底芯片中，在标准内存读取路径上执行内联反量化，返回反量化后的权重直接供张量核心使用。
- **关键技术细节**：
  - **边带标签（Sideband Tag）**：每个内存读取请求附加 3 位边带标签，指示反量化模式（如 INT4→FP16）或旁路（bypass），不改变有效地址，无需 GPU ISA 变更。
  - **区域查找表（Region Lookup Table）**：GPU 侧运行时维护一组小表（约 3.5~3.9 KB），根据虚拟地址匹配注册的量化权重区域并生成标签。
  - **伪通道感知布局（Pseudo-Channel-Aware Layout）**：将量化权重组及其对应的 S/Z 元数据重新组织到同一伪通道内，避免跨伪通道通信；必要时对 S/Z 进行少量复制（开销可忽略）。
  - **DQB 微架构**：
    - **标签解析器 + 标签历史缓冲区**：解析标签，存储模式信息指导后续处理。
    - **S/Z 请求生成器 + 本地 S/Z 缓存**：根据权重地址计算组索引，从缓存或内存获取 S/Z，缓存命中率高（S/Z 读取仅占总读取的 1~7%）。
    - **数据处理块（Data Processing Block）**：共享 FP32 ALU 执行反量化算术，支持多种格式转换。对 FP→FP 转换使用导线映射（wire-mapping）和轻量多路选择器；对 INT→FP 转换使用共享 LUT（8-bit→FP32 只存储高 16 位，零填充优化）。
  - **流程**：GPU 发出带标记载入请求 → DQB 解析标签 → 若缺失则发起 S/Z 读取 → 权重数据到达后即时反量化 → 反量化结果返回 GPU 正常加载响应路径。
- **公式**：反量化操作沿用标准公式 \( \text{Dequant}(x) = (x - z) \cdot s \)。

## 7. 优点
- **创新性**：首次将反量化卸载到 HBM 基底芯片中，利用自定义 HBM 趋势，避免 GPU 核心修改。
- **轻量集成**：仅需 3 位边带标签和区域查找表，不改变 GPU 计算路径、指令集或缓存一致性。
- **解决传统 NMP 难题**：通过伪通道感知布局和 S/Z 同地址空间，规避了地址翻译和数据跨通道放置两大挑战。
- **高效格式支持**：导线映射和共享 LUT 实现低开销的多格式转换，易于扩展未来低精度格式。
- **实验严谨**：模拟器校准真实 GPU，覆盖主流模型和批次大小，提供性能、能耗、面积、热等多维度验证。

## 8. 不足与局限
- **依赖自定义 HBM**：当前未商用，需等 HBM4 等定制基底芯片技术成熟。
- **模型规模局限**：仅评估 7B/8B 级模型，未验证对更大 LLM（如 70B+）的有效性。

- **单 GPU 评估**：未考虑多 GPU 分布式推理场景下的负载均衡和扩展性。
- **小批量场景限制**：小 batch size 时性能略逊于 AWQ-v2 的融合内核，说明 StreamDQ 并非适用于所有推理场景。
- **不支持量化激活**：仅处理权重反量化，无法加速激活量化的 W8A8 等全量化推理。
